# Format: docs/impl-notes.html Created when a task is driven by a `` block. Updated in-place throughout implementation. One file per spec — never append multiple specs into one file. ## When to create When the user message contains a `` block. Extract the spec content, seed the HTML below, then implement. Update Decision/Deviation/Tradeoff/Question entries as you work. Resolve or explicitly flag all open questions before closing. ## HTML template Produce exactly this structure. Fill placeholders marked {{LIKE_THIS}}. Inline all styles — no external dependencies. File must open standalone in a browser. ```html impl-notes — {{SPEC_TITLE}}

Implementation Notes — {{SPEC_TITLE}}

{{SESSION_ID}} {{DATE}} docs/impl-notes.html {{STATUS}}

Spec

{{SPEC_CONTENT}}

Design Decisions

No decisions recorded yet.

Deviations

No deviations recorded yet.

Tradeoffs

No tradeoffs recorded yet.

Open Questions

No open questions.

``` ## Entry field reference | Field | What to write | |-------|--------------| | `{{SPEC_TITLE}}` | One-line title from the spec or user message | | `{{SESSION_ID}}` | `$CLAUDE_SESSION_ID` or "unknown" | | `{{DATE}}` | ISO datetime when file was created | | `{{STATUS}}` | `in-progress` → `complete` → `blocked` | | `{{SPEC_CONTENT}}` | Full spec text verbatim, unmodified | | `{{TIMESTAMP}}` | ISO datetime of the entry | ## Update rules - **Create** when you see `` in the user message — extract content, seed file, begin work - **Update in-place** — this is not append-only like plan.md; edit the existing sections - **Remove `.empty` placeholders** as you add real entries to each section - **Resolve questions** by changing `badge-open` → `badge-resolved` and adding the answer inline - **Set status to `complete`** in the header meta when implementation is done and all questions resolved or explicitly deferred
