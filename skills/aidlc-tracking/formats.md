# AIDLC Tracking — Canonical File Formats

Load this file before creating or appending to any tracking file.

---

## tasks/tracker.md

```markdown
# Task Tracker
<!-- Append-only. Newest at TOP. Two entry types: task-complete and pre-compact. -->
<!-- Format: ## YYYY-MM-DD HH:MM:SS — <summary> -->
```

### Entry type 1 — Task complete

```markdown
## YYYY-MM-DD HH:MM:SS — <one-line task summary>
**Type:** task-complete
**Outcome:** <what was achieved>
**Files changed:** <comma-separated list, or "none">
```

### Entry type 2 — Pre-compact snapshot

```markdown
## YYYY-MM-DD HH:MM:SS — [Pre-Compact Snapshot]
**Type:** pre-compact
**Working on:** <current task or goal>
**Completed this session:**
- <item>
- <item>
**Key decisions:**
- <decision and rationale>
**Files changed:** <comma-separated list>
**Open / in-progress:** <what is unfinished>
**Resume from:** <exact state — enough detail for a cold-start in the next session>
```

## tasks/lessons.md

```markdown
# Lessons
<!-- Append-only. Newest at TOP. Per-repo scope. -->
<!-- Format: ## YYYY-MM-DD — <title> -->
```

### Lesson entry

```markdown
## YYYY-MM-DD — <short title>
**Rule:** What to do (or not do) — stated as an actionable instruction
**Why:** What went wrong or what was learned that motivated this rule
**Trigger:** The condition or context when this rule applies
**Applies to:** <file path / module / pattern / language / tool>
```

## docs/plan.md

```markdown
# Plan Log
<!-- Append-only. Newest at TOP. Written BEFORE implementation begins. -->
<!-- Format: YYYY-MM-DD HH:MM:SS - <one line summary> -->
```

### Plan entry

```markdown
---

YYYY-MM-DD HH:MM:SS - <one-line summary of what is being planned>

### Context
<Why this is being done. What triggered this work. Stakeholder input or constraints.>

### Approach
<What will be done and why this approach was chosen over alternatives.>

### Tradeoffs
<What is being traded off. What is explicitly out of scope. Known risks.>

### Checklist
- [ ] <step>
- [ ] <step>
```

## docs/design-review.md

```markdown
# Design Review Log
<!-- Append-only. Newest at TOP. Generated by /design-review + review-council skill. -->
<!-- Format: ## YYYY-MM-DD HH:MM:SS — <what was reviewed> -->
```

### Design review entry

```markdown
---

## YYYY-MM-DD HH:MM:SS — <one-line description of what was reviewed>

### Scope
<What was reviewed, why now, and what triggered this review.>

### Phase 0 — Human Brief
<User's stated concerns, focus areas, and known constraints from the opening dialogue.>

### Council Composition
| Persona | Domain | Model |
|---------|--------|-------|
| <name>  | <domain> | <sonnet/haiku> |

### Round 1 — Independent Assessments

#### <Persona Name> (<model>)
**Stance:** proceed | caution | block
**Top findings:**
- <finding>
**Blocker (if any):** <blocker description, or "none">
**Questions to council:**
- <question>

### Round 2 — Council Debate
<Each persona's response to key challenges raised. Note where positions changed or held.>

### Converged Concerns
<Issues multiple personas flagged independently — highest signal. List each with which personas agreed.>

### Blocking Concerns
<Issues that must be resolved before proceeding. For each: what must change and why.>
<If human overrode a blocker: record the override and stated rationale explicitly.>

### Phase 5 — Human Input
<User's responses to council questions. Any blocker overrides recorded here with rationale.>

### Final Synthesis
<Conclusion incorporating all council input and human input.>

### Decision
- [ ] Proceed as-is
- [ ] Proceed with modifications: <list modifications>
- [ ] Redesign required: <list blocking concerns to address first>

### Action Items
| Item | Owner | Priority |
|------|-------|----------|
| <item> | <persona domain or "human"> | high / medium / low |
```
