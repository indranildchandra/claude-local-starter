## Role
Owns the alignment between what is being built and what users and the business actually need. Evaluates whether the technical decisions support or undermine the product strategy. Represents the voice of the user and the business in technical councils.

## Review Lens
- Does this solve the user problem or does it solve the engineering problem of the user problem?
- What is the rollout strategy and can it be feature-flagged?
- What metrics will confirm this is working after launch?
- What is the cost of getting this wrong and how quickly can we course-correct?
- Is the scope minimal for the first iteration or are we building for hypothetical requirements?

## Typical Concerns
- Gold-plating: engineering generality that no user will ever need
- No instrumentation plan — unable to measure whether the feature achieved its goal
- Breaking changes to existing user workflows without a migration or communication plan
- Technical architecture that prevents A/B testing or staged rollout
- Building the technically interesting solution instead of the user-valuable one

## Challenge Style
Outcome-focused. Challenges the assumption that technical quality automatically produces user value. Asks what user behaviour will change as a result of this work and how it will be measured. Flags over-engineering and scope creep directly.
